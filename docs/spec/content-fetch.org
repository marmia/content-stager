* Content Fetch（URL→本文取得）

** ゴール

- URLから本文テキストを抽出し、後続のTopic分解に渡せる状態にする。

** MVP方針

- 本文取得は既存の Agent Skills を利用する。
- MVPでは agent-browser（CLI）を使用する。
- 将来的に web-reader-article も使用できるようにする（プロバイダ切替）。

** 抽象化（プロバイダ切替のための契約）

本文取得は、以下の「共通出力」を満たすこと。

- 必須出力:
  - url
  - fetched_at
  - provider（例: agent-browser / web-reader）
  - status（ok / failed）
  - article_text（本文テキスト。failed以外では空にしない）
- 任意出力:
  - title（推測でよいが、未取得なら空）
  - language（推測でよいが、未取得なら空）
  - warnings（除去できなかったノイズ等）
  - error（failed時の要約。機密は含めない）

** 失敗時の扱い

- failed の場合:
  - ContentはS1に留める（S2へ自動遷移しない）
  - 再実行（同一URLでのリトライ）が可能であること

** 「不完全な本文」の扱い（MVPの前提）

- fetch段階では partial のような強い分類を前提にしない（warningsで表現する）。
- 取得した本文は可能な限りTopic分解を試みる。
- 本文が不完全で意味をなさない場合はTopicを抽出せず処理を終了し、S1に留めて再実行可能にする。

** 決定性（重要）

- 本文取得の結果が揺れる可能性は受け入れるが、
  その揺れが「状態遷移」を直接引き起こさない設計にする。
- プロバイダ選択は明示（設定/引数/環境変数）で固定できること。
